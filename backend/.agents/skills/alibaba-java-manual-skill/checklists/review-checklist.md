# 使用检查清单

> 本清单只用于触发审查动作，不替代原文。涉及具体判断时，优先查 `references/manual-full-text.md` 中对应章节原文。

## 审查优先级

1. 先检查【强制】：命名、代码格式、OOP、日期时间、集合、并发、异常日志、安全、MySQL、工程结构、设计规约。
2. 再检查【推荐】：可读性、可维护性、性能、测试、日志、索引、应用分层等。
3. 最后参考【参考】：结合项目上下文给出建议，不作为硬性阻断。

## 输出格式建议

审查代码时按以下结构输出：

- 结论：通过 / 需要修改 / 存在风险
- 命中规约：章节 + 【强制/推荐/参考】+ 原文关键词
- 问题位置：文件、类、方法、行号（如可得）
- 修改建议：保持最小改动，不重写业务逻辑
- 示例代码：只展示必要片段

## 不能做的事

- 不要把【推荐】说成【强制】。
- 不要引入手册没有明确要求的规则，并声称来自阿里巴巴手册。
- 不要为了“更现代”而改写原文规约含义。
- 不要在没有上下文时强行要求项目采用某个框架或目录结构。
